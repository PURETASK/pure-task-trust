# Phase C (shipped 2026-04-30)

## useCurrentProfile primitive
- File: `src/hooks/useCurrentProfile.ts`
- Two flavours:
  - `useCurrentProfile()` — hook returning `{ userId, role, clientProfileId, cleanerProfileId, isLoading, error }`
  - `getCurrentProfileIds(queryClient, userId)` — async helper for use inside `useMutation` callbacks (cached against same TanStack key)
- **Rule**: Never re-write `supabase.from('client_profiles').select('id').eq('user_id', user.id)` (or cleaner equivalent). Use the primitive.
- Swept consumers: `useJob.ts`, `useJobCheckins.ts`. The remaining `client_profiles`/`cleaner_profiles` reads in admin/analytics hooks are legitimate (fetch ALL profiles, not the current user's id).

## Database security
- Linter went 35 → 15 warnings via one migration revoking EXECUTE from anon+authenticated for 11 SECURITY DEFINER functions:
  - Vault: `vault_secret_exists`, `vault_insert_cron_secret`
  - Email queue: `enqueue_email`, `delete_email`, `read_email_batch`, `move_to_dlq`
  - Trigger-only: `handle_new_user`, `handle_agent_ticket_message`, `guard_jobs_financial_writes`, `guard_credit_account_writes`, `update_updated_at_column`, `validate_cleaner_hourly_rate`
  - Codegen: `generate_referral_code`
- **Kept callable** by authenticated (each enforces own auth internally): `has_role`, `get_user_role`, `is_ticket_owner`, `cleaner_has_job_with_client`, `client_has_job_with_cleaner`, `log_admin_action`, `create_booking_atomic`, `approve_job_atomic`. Do NOT switch these to SECURITY INVOKER — would break RLS recursion.
